# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **Google ADK** (Agent Development Kit) multi-agent system that produces complete Google Ads / SEA (search engine advertising) campaigns: landing-page analysis → strategy → search-intent → keyword research → ad copy → Excel export, plus a separate optimizer sub-team for KPI-driven tuning. All agent instructions and all generated output are in **German** — keep them German when editing.

Generated by `agents-cli` (Google Agents CLI) from the `adk` base template. `GEMINI.md` is the original agent-guidance file for this scaffold and carries operational rules (see "Conventions" below); `agents-cli-manifest.yaml` records scaffold settings (`deployment_target: none`, `session_type: in_memory`).

## Commands

Toolchain is **uv** (deps) + **agents-cli** (dev/eval/deploy). Install: `uv tool install google-agents-cli`.

| Command | Purpose |
|---------|---------|
| `agents-cli install` | Install/sync dependencies (uv-based) |
| `agents-cli playground` | Local web playground, auto-reloads on save |
| `agents-cli lint` | ruff + ty (type check) + codespell |
| `agents-cli eval run` | Run evalsets in `tests/eval/` |
| `uv run pytest tests/unit tests/integration` | Run unit + integration tests |
| `uv run pytest tests/unit/test_exporter.py` | Run a single test file |
| `uv run pytest tests/unit -k exporter` | Run tests matching a name |
| `uv run adk ...` | Raw ADK CLI |
| `agents-cli deploy` | Deploy to dev (**requires explicit human approval**) |

Local dev (the way Florian actually runs it day-to-day):

| Command | Purpose |
|---------|---------|
| `.venv/bin/python -m uvicorn app.fast_api_app:app --host 127.0.0.1 --port 8002 --reload` | Run backend locally on **:8002** (not 8001 — that port is reserved for another local project) |
| `cd ../ads-ai-agent && npm run dev` | Run the React frontend on **:5180** with Vite, which proxies API routes to `127.0.0.1:8002` |

- **Unit tests** (`tests/unit/`, 41 cases) run fully offline, no GCP credentials needed (e.g. `test_exporter.py` exercises `save_campaign_as_excel` directly).
- **Integration tests** (`tests/integration/`) and **evals** call the real Gemini model, so they need an API key in `.env`.

## Architecture

### Repo layout after the agents/<name>/ refactor

The codebase was refactored from a 2300-line `app/agent.py` monolith into per-agent folders. Code and the agent's domain skill (markdown playbook) live together so editing the Strategy agent doesn't require jumping between `app/agent.py` and a separate `app/skills/` folder.

├── common.py                                # shared: load_dotenv, mongodb_toolset, GLOBAL_RULES, apply_rules,
│                                            #   mongodb_persist_directive, json_block_directive, cached_skill_from
├── fast_api_app.py                          # FastAPI app + ADK routes + /conversations endpoint
├── google_search.py                         # google_autocomplete tool (used by search_intent agent)
├── google_ads_client.py / oauth / settings  # campaign_builder tools
├── conversation_store.py                    # chat persistence (filesystem-backed)
├── app_utils/                               # telemetry config
└── agents/
    ├── __init__.py                          # re-exports all factories
    ├── team_lead/team_lead.py               # root coordinator (sea_team_lead) + TEAM_LEAD_MEMORY_CHECK
    ├── landing_page/landing_page.py         # LP_OUTPUT_SCHEMA + tools (fetch, cache helpers)
    ├── strategy/
    │   ├── strategy.py                      # STRATEGY_OUTPUT_SCHEMA + factory
    │   └── skills/strategy_skill.md         # 3-layer model playbook (Marke→Produkt→Lösung + NO-COMPETITOR-POLICY)
    ├── search_intent/search_intent.py
    ├── keyword/
    │   ├── keyword.py                       # KEYWORDS_OUTPUT_SCHEMA + factory
    │   └── skills/keyword_skill.md          # scoring rubric, 5-layer negatives, ad-group rules per Schicht
    ├── copywriter/
    │   ├── copywriter.py
    │   └── skills/copywriter_skill.md       # 3-pillar RSA blueprint, voice-pool binding, no-competitor rules
    ├── translator/translator.py
    ├── excel_exporter/excel_exporter.py     # save_campaign_as_excel + factory (writes data/exports/*.xlsx)
    ├── campaign_builder/campaign_builder.py # list_google_ads_accounts + create_search_campaigns + factory

`app/agent.py.backup` is the pre-refactor 2300-line monolith, kept around as reference. `app/skills/` (3 .md + a loader) is the pre-refactor skills folder; nothing imports it anymore but it's kept as a backup. Tests import historic names from `app.agent` — those still work via the re-export block at the bottom of `app/agent.py`.

### Agent hierarchy

`root_agent` (`sea_team_lead`) coordinates 9 sub-agents through ADK's `sub_agents=[...]`:

```
sea_team_lead (root coordinator)
├── landing_page_agent      # scrapes URL, extracts USPs/audience/tone, CACHES result
├── strategy_agent          # 3-layer campaign structure + budget/bidding recommendations
├── search_intent_agent     # live Google Autocomplete per seed, clusters by intent stage
├── keyword_agent           # keyword clusters with match types + 5-layer negatives
├── copywriter_agent        # up to 15 headlines (≤30 chars) + 4 descriptions (≤90) per cluster, RSA
├── translator_agent        # translates ad copy, preserving char limits
├── optimizer_team_lead     # sub-coordinator for KPI optimization (5 sub-agents):
│   ├── quality_score_optimizer
│   ├── ctr_booster_optimizer
│   ├── conversion_optimizer
│   ├── optimizer_keyword_agent
│   └── optimizer_searchterms_agent
├── excel_exporter_agent    # converts everything to semantic JSON for save_campaign_as_excel
└── campaign_builder_agent  # actually creates the search campaigns in Google Ads (paused, 1 EUR/day)
```

The team lead encodes the canonical **end-to-end workflow** (account briefing → analyze → strategy → search-intent → keywords → ask the user for headline structure → write copy per cluster → merge → Excel export). Step "ask for headline structure" is interactive: the `copywriter_agent` may stop and ask the user how headlines should be structured; the root forwards that question verbatim and pauses.

Every agent is built by a `create_*_agent()` factory in its folder, uses model `gemini-flash-latest` with 3 retries, and wraps its instruction in `apply_rules()` from `app/common.py`.

### Account-briefing flow (team_lead)

Before any URL workflow runs, the team_lead asks five questions and emits a `account_briefing` JSON block in chat:
1. Context (brand/product/URL)
2. Conversions/month (drives `account_phase: "new"|"ramping"|"mature"`)
3. Daily budget (EUR)
4. Primary goal (ROAS / CPA / revenue_growth / brand_awareness) + `target_value`
5. **Which of the 3 default layers (Brand Pure / Brand+Produkt / Generic) already run** → `existing_layers: list[str]`

The briefing is read by every downstream agent. **Update both the team_lead briefing prompt and the schema in lockstep** if you change it.

### Three-layer strategy model (Marke → Produkt → Lösung)

`strategy_agent` always builds **three campaigns** by default:

| Schicht | `purpose` | `voice_pool` | Final-URL | Match-Type | Budget-Default |
|---|---|---|---|---|---|
| **Brand Pure** | `Brand` | `brand` | Homepage/Shop | EXACT-dominant | 10–15 % |
| **Brand + Produkt** | `Brand` | `product` | Produkt-LP | EXACT-dominant | 15–25 % |
| **Generic / Non-Brand** | `Generic` | `product` | Produkt-LP | PHRASE-dominant | 60–75 % |

Layers listed in `existing_layers` are skipped (budget reallocates, tilted toward Generic). Pro übersprungener Schicht ist ein konkreter Advisory-Satz in Prosa-Sektion 6 Pflicht.

**Hard policies that constrain ALL agents:**
- **NO-COMPETITOR-POLICY** — never build dedicated competitor campaigns / `vs <X>` ad-groups / `purpose: "Competitor"`. Competitor names from `landing_page_analysis.competitors` go into `recommended_negative_keywords` only. Reasons: high CPC, weak CR, trademark risk, defensive headlines. Enforced in `strategy/skills/strategy_skill.md` Kapitel 5 + `keyword/skills/keyword_skill.md` Kapitel 3 + `copywriter/skills/copywriter_skill.md` Kapitel 4.
- **Naming convention** (hart): every `campaigns[].name` and `ad_groups[].name` MUSS einen der drei Präfixe tragen — `Brand Pure – …`, `Brand+Produkt – …`, `Generic – …` (also `Generic Search – …` ist OK). Ohne Präfix scheitert der Excel-Export.
- **`use_in_copy` flag on keywords** — each keyword has `use_in_copy: bool`. `true` keywords are **Headline-Anchors** the copywriter pulls from; `false` keywords are **Bidding-Only** (e.g. retail modifiers like "dm", research tails like "test"). Per ad-group 3–7 anchors required. Defined in `keyword_skill.md` Kapitel 5.

### Global behavior rules
`GLOBAL_RULES` + `apply_rules()` in `app/common.py` append four hard constraints to **every** agent instruction: no emojis, no preamble/filler, no leaked credentials, and **always use real German umlauts (ä/ö/ü/ß), never the ASCII replacements (ae/oe/ue/ss)** — also inside JSON-block values.

Preserve this when adding or editing agents; create them through the same factory pattern and wrap instructions with `apply_rules()`.

### Skill files (per-agent domain playbooks)

Three agents have a dedicated `skills/<name>_skill.md` markdown playbook loaded via `cached_skill_from(SKILL_PATH)` at agent-construction time (re-read on backend reload):
- `agents/strategy/skills/strategy_skill.md` — universal 3-layer model, existing_layers skipping logic, ad-group rules per Schicht, NO-COMPETITOR-POLICY
- `agents/keyword/skills/keyword_skill.md` — match-type philosophy + 1–10 scoring rubric, 5-layer negatives (Strategy starter / Search-Intent signals / Score 1–4 / LP-Gap-Analyse / Awareness), `use_in_copy` carve-out, drei-Schicht ad-group keyword rules
- `agents/copywriter/skills/copywriter_skill.md` — 3-pillar RSA blueprint, voice-pool binding, tonality mapping, NO-COMPETITOR-POLICY

When editing a skill, the next backend restart picks up changes automatically.

### Persistence: two backends, chosen at import time
`landing_page_agent` caches customer profiles to avoid re-scraping. The backend is decided when `app/agents/landing_page/landing_page.py` is imported, based on env var `MDB_MCP_CONNECTION_STRING`:
- **Set** → a MongoDB MCP toolset (`npx @mongodb-js/mongodb-mcp-server`, requires Node/npx) is attached; the agent's instruction tells it to `connect` first, then `find`/`insert-many` in db `sea_team_lead`, collection `customer_profiles`.
- **Unset** → falls back to local JSON files via `save_customer_profile`/`load_customer_profile`, written to `data/customers/<sanitized-domain>.json`.

The agent's tool list **and** instruction text differ between these two paths — edit both branches together. MongoDB persistence is also extended by `strategy_agent`, `keyword_agent`, and `copywriter_agent` via `mongodb_persist_directive(agent_field)` from `app/common.py`.

### Excel export (`save_campaign_as_excel`)
Writes styled `.xlsx` to `data/exports/`. It clones header rows, column widths, merges, and cell styling from the template workbook `data/template/adsforsocial_analyse.xlsx` (sheets: Landingpage-Analyse, Kampagnenstruktur, Keyword-Recherche, Anzeigentexte (RSA)), then fills data into fixed cell ranges. If the template is missing it draws a fallback banner layout. A legacy "old format" branch (`is_old_format`) handles a plain `{sheet_name: [rows]}` dict for backward compatibility. The hard-coded color palette and fixed-row slots (e.g. USPs rows 6–11) are tied to the template layout — changing one means changing the other. Lives at `app/agents/excel_exporter/excel_exporter.py`.

### Model provider selection
`app/common.py` (and `app/fast_api_app.py`) load `.env` and map `VITE_GEMINI_API_KEY` → `GEMINI_API_KEY` (the `.env` is shared with the Vite frontend). If a Gemini/Google API key is present, `GOOGLE_GENAI_USE_VERTEXAI=False` (AI Studio API); otherwise it falls back to Vertex AI with `GOOGLE_CLOUD_LOCATION=global`.

### Entry points
- ADK runtime: `app.agent:root_agent`, exposed as `App(name="app")` and re-exported from `app/__init__.py`.
- HTTP server: `app/fast_api_app.py` builds the ADK FastAPI app (in-memory sessions, optional GCS artifact bucket + OTel→Cloud telemetry). The `Dockerfile` runs `uvicorn app.fast_api_app:app` on port 8080 (prod). Locally we run on :8002. Adds a `/feedback` endpoint and a `/conversations` endpoint backed by `conversation_store.py`.
- Telemetry: `app/app_utils/telemetry.py` configures GenAI/OTel export to GCS (metadata-only by default).

### Frontend
The real frontend is the standalone **`ads-ai-agent/`** project (sibling folder, React 19 + Vite 6 + Tailwind 4), served at http://localhost:5180/ in dev. Vite proxies API routes to `127.0.0.1:8002` (see `ads-ai-agent/vite.config.ts`, `BACKEND` constant). UI conventions:
- Agent avatars live in `ads-ai-agent/public/*.jpg`, mapped in `ads-ai-agent/src/agents.ts` via `avatarUrl`
- Avatar border colors match the agent's text accent (`borderColorClass` in the same registry)
- The visual-card bubble fades in via `.animate-fade-in-up` (see `ads-ai-agent/src/index.css`) so the second bubble doesn't pop in when the JSON block lands at the end of a stream
- The ThinkingIndicator uses a radar-style ping in the agent's accent color + a soft text pulse

The old scaffold frontend at `sea-team-lead/frontend/` was deleted during cleanup; references in `pyproject.toml` and the FastAPI `StaticFiles` mount were removed.

## Conventions (from GEMINI.md)

- **Do not change the `model` value** of any agent unless explicitly asked.
- Model **404 errors** → fix `GOOGLE_CLOUD_LOCATION` (use `global`), not the model name.
- **ADK tool imports**: import the tool *instance*, not the module — e.g. `from google.adk.tools.load_web_page import load_web_page`.
- Run Python via `uv run python ...` (after `agents-cli install`).
- If the same error recurs 3+ times, fix the root cause instead of retrying.
- Per-agent files start with `# ruff: noqa` — lint is disabled there; ruff config (`pyproject.toml`) sets line-length 88 and ignores E501/C901/B006.

## Workflow conventions

- **Skills are domain playbooks**, not behaviour rules. Hard rules (JSON schema, char limits, naming convention, END-CHECK) live in the Python instruction; skills inform *content*. On conflict, hard rules win — every skill says this in its header.
- **Editing a skill**: edit the agent's local `skills/*.md`, restart the backend, the change is live. Don't edit `app/skills/` (backup-only, not loaded).
- **Adding a new agent**: create `app/agents/<name>/<name>.py` with a `create_<name>_agent()` factory + `__init__.py` exporting it; if it has a skill, add `app/agents/<name>/skills/<name>_skill.md` and load via `cached_skill_from(SKILL_PATH)` from `app/common.py`. Then add the import + sub-agent registration to `app/agents/team_lead/team_lead.py` and re-export from `app/agents/__init__.py`. Update tests that import from `app.agent` via the backwards-compat block in `app/agent.py`.
- **Single-File-Refactor Trigger**: when an instruction block in a `<name>.py` agent file grows past ~300 lines of pure text, consider extracting the prose into a skill markdown file.
